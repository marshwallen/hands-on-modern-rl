# 第 12 章：Agentic RL——工具调用、多轮交互与智能体训练

> **本章目标**：理解如何用 RL 训练能够调用工具、与环境多轮交互的智能体。这是 2025-2026 年 RL 在大模型领域最热门的应用方向。
>
> **本章地位**：前几章的 RL 都是"单轮"的——输入 prompt，输出 response，打分。但真实的 Agent 需要连续多步行动：搜索信息、执行代码、观察结果、调整策略。本章将 RL 从"单轮对话"扩展到"多轮决策"，这是 Agentic AI 的核心基础设施。

## 12.1 从单轮 RL 到多轮 RL

### 单轮 vs 多轮：本质区别

在前面的章节中，我们处理的 RL 问题都是单轮的：

```
单轮 RL（前面所有章节）：
  Prompt → 模型生成完整回答 → Reward Model 打分 → 更新策略
```

但一个真正的 Agent 的工作方式完全不同：

```
多轮 RL（Agentic 场景）：
  用户提问 → 模型调用搜索工具 → 观察搜索结果 → 模型生成代码 → 执行代码 → 观察报错
  → 模型修复代码 → 再执行 → 成功 → 返回最终答案 → 基于最终结果给 Reward
```

**核心差异**：

- **动作空间扩展**：模型不再只生成文本，还需要决定"何时调用工具"、"调用哪个工具"
- **奖励延迟**：最终 Reward 在多步交互之后才出现，中间步骤没有直接反馈
- **信用分配更难**：第 2 轮的一个错误调用导致第 7 轮的最终失败，怎么归因？

### Agentic MDP 建模

一个 Agentic 系统可以建模为特殊的 MDP：

- **状态 $s_t$**：对话历史 + 工具返回结果 + 环境当前状态
- **动作 $a_t$**：生成文本 / 调用工具 A / 调用工具 B / ...
- **转移 $P(s_{t+1}|s_t, a_t)$**：工具执行的结果是不可控的环境动态
- **奖励 $r(s_t, a_t)$**：中间步骤 reward = 0，最终 success/failure 给 1/0

这比标准 LLM 的 RL 复杂得多——标准 LLM 只有一个动作（生成 token），而 Agent 有多个异构动作。

## 12.2 工具调用 RL（Tool-Use RL）

### 为什么需要 RL 来训练工具调用？

监督学习（SFT）可以教会模型"工具调用的格式"，但教不会"何时该调用工具"：

|        | SFT                        | RL                                     |
| ------ | -------------------------- | -------------------------------------- |
| 学什么 | 工具调用的语法格式         | 何时调用、调哪个、如何组合             |
| 数据   | 需要人工标注的工具调用轨迹 | 只需要最终结果（成功/失败）作为 reward |
| 泛化   | 只能处理见过的工具组合     | 能探索新的工具使用策略                 |

### 核心方法

**ReTool**（Reasoning with Tools）：让模型在推理过程中自由调用计算器和代码解释器，用 RL 优化"何时调用"的策略。

**VERL-TOOL**：跨领域工具调用 RL 训练框架，覆盖数学推理、SQL 生成、Web 搜索、软件工程等场景。

**ToolRL**：将工具调用视为 RL 中的一个特殊动作，扩展策略的动作空间。

### 奖励设计

工具调用的 reward 不像偏好对齐那样主观，可以基于客观信号：

| 场景     | Reward 来源          | 类型                |
| -------- | -------------------- | ------------------- |
| 代码生成 | 代码是否通过单元测试 | 二元（0/1）         |
| 数学推理 | 计算器结果是否正确   | 二元（0/1）         |
| Web 搜索 | 是否找到正确答案     | 二元 + 路径效率惩罚 |
| SQL 生成 | 查询结果是否匹配     | 二元 + 执行时间惩罚 |

这实际上就是第 8 章提到的 **RLVR（可验证奖励）** 在 Agentic 场景的直接应用。

## 12.3 多轮交互 RL（Multi-Turn RL）

### 核心挑战：跨轮次的信用分配

多轮 RL 中最困难的数学问题是**信用分配（Credit Assignment）**：

- 7 轮交互，最终失败了
- 是第 2 轮的搜索词写错了？还是第 5 轮的代码有 bug？
- Reward 只在最后一轮给出，中间轮次没有反馈

### 两种奖励方案

**ORM（Outcome Reward Model）**：只看最终结果

```
问题 → 思考₁ → 调用工具 → 思考₂ → 调用工具 → ... → 最终答案 → 对/错？
                                                                      ↑
                                                              只有1个reward信号
```

**PRM（Process Reward Model）**：每一步都评估

```
问题 → 思考₁ → 调用工具 → 思考₂ → 调用工具 → ... → 最终答案
        ↓          ↓          ↓          ↓                ↓
      对吗?      对吗?      对吗?      对吗?            对吗?
        ↑          ↑          ↑          ↑                ↑
                                                    每步都有reward信号
```

PRM 的优势：更密集的学习信号，能指导中间步骤的改进方向。
PRM 的代价：需要标注每一步的正确性，数据成本极高。

### 代表性框架

**MLMT-RL**（Multi-Level Multi-Turn RL）：在多个粒度上分配 reward，比单层 GRPO 高 14 个百分点。

**Verlog**（CMU）：处理变长 episode 的多轮 RL 框架，支持 turn-level reward + discounting。

### Turn-Level Discounting

多轮 RL 中一个关键技术是对不同轮次的 reward 做折扣：

$$R = \sum_{t=1}^{T} \gamma^t \cdot r_t$$

其中 $\gamma < 1$ 表示远期 reward 的权重降低。直觉上：越早犯的错误，对最终结果的影响越大（因为后续步骤都建立在前面的基础上）。

## 12.4 Web Agent RL 与 Code Agent RL

### Web Agent RL

训练能够浏览和操作网页的 Agent：

- **动作**：点击、输入文本、滚动、导航
- **状态**：网页截图 + DOM 树
- **Reward**：任务完成度（是否找到正确信息、是否成功提交表单）
- **挑战**：网页环境的巨大状态空间、动态内容加载

### Code Agent RL

训练能够写代码、调试、迭代的 Agent：

- **动作**：生成代码、执行代码、阅读报错、修改代码
- **状态**：代码文件 + 执行输出 + 错误信息
- **Reward**：测试通过率 + 代码质量
- **关键技术**：ICML 2025 证明"单步 reward 可以有效引导多轮代码生成"

## 12.5 Agentic RL 的工程挑战

### 环境构建

Agentic RL 的最大工程瓶颈是**环境**——需要一个能可靠地执行工具调用并返回结果的系统：

```
┌─────────────────────────────────────────┐
│          RL Training Loop               │
│                                         │
│  ┌──────┐   ┌──────────┐   ┌────────┐  │
│  │ Actor │ → │ 工具执行器 │ → │ Reward │  │
│  │(LLM)  │   │ (沙箱)    │   │ Service│  │
│  └──────┘   └──────────┘   └────────┘  │
│       ↑                              │  │
│       └──────────────────────────────┘  │
└─────────────────────────────────────────┘
```

关键要求：

- **安全性**：代码执行必须在沙箱中（防止模型执行恶意代码）
- **可复现**：相同输入应该返回相同结果（否则训练不稳定）
- **低延迟**：工具调用的响应时间直接影响训练吞吐

### RL 基础设施

Agentic RL 的训练基础设施比标准 LLM RL 复杂得多：

| 组件         | 标准 LLM RL           | Agentic RL                             |
| ------------ | --------------------- | -------------------------------------- |
| Rollout      | GPU 生成文本          | GPU 生成文本 + **CPU 执行工具**        |
| 环境交互     | 无                    | **需要工具沙箱、Web 环境、代码执行器** |
| Reward       | Reward Model 打分     | **环境执行结果**（代码通过/失败）      |
| Episode 长度 | 固定（生成 token 数） | **可变**（不同任务需要不同轮数）       |

### 代表性框架

- **AWorld-RL**：完整的 Agentic RL 训练环境
- **NeMo Gym**（NVIDIA）：科学 Agent 的 RL 训练基础设施
- **Agentic RL Training Recipes**：开源训练方案集合

## 12.6 回看实践

### 本章关键收获

1. **Agentic RL = 多轮 RL + 工具调用**：从"生成文本"扩展到"在环境中行动"
2. **信用分配**是核心难题：ORM 简单但信号稀疏，PRM 密集但标注昂贵
3. **RLVR 是天然选择**：工具执行结果是客观可验证的，不需要训练 Reward Model
4. **环境是工程瓶颈**：安全、可复现、低延迟的工具沙箱是 Agentic RL 的基础设施
