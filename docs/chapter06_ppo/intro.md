# 第 6 章：PPO——稳定训练的艺术

> **本章目标**：深入理解当前使用最广泛的 on-policy 算法 PPO。你将掌握裁剪机制和 GAE 两大核心组件，并理解 PPO 如何从游戏控制延伸到 LLM 对齐。
>
> **本章暗线**：PPO 需要 Critic 网络来估计优势（接第 5 章），但在 LLM 场景中，还需要一个更关键的东西——**Reward Model（奖励模型）**。人类无法给每个回答写一个精确的数字分数，只能给出"回答 A 比回答 B 好"的偏好判断。Bradley-Terry 模型把这些偏好转成了一个打分器。**理解了"训练一个 RM 有多难"，下一章 DPO 的震撼感才会拉满。**

- 熵（Entropy）下降意味着什么？——从"到处探索"到"专注好动作"

- **你将思考**：RL 训练的稳定性不是免费的，需要精心的算法设计

## 5.3 理论：策略更新的危险

- 朴素策略梯度的问题：一步更新太大 → 策略崩溃（忘了之前学的一切）
- 重要性采样（Importance Sampling）：用旧策略收集的数据来更新新策略
- 策略比率 r_t(θ) = π_new(a|s) / π_old(a|s)——衡量新旧策略的差异
- **你将理解**：为什么不能无限制地重复利用旧数据更新策略

## 5.4 TRPO 与 PPO：信任域的两代解法

- **核心问题**：策略更新多大才算"安全"？——需要一个数学约束来回答
- **TRPO（Trust Region Policy Optimization，2015）**：PPO 的直接前身
  - 约束：每次更新后，新旧策略的 KL 散度必须小于 δ（通常 0.01）
  - 直觉："新策略不能离旧策略太远"——用 KL 散度严格丈量"距离"
  - 方法：二阶优化（需要计算 Hessian 矩阵），理论上优雅但**计算极其昂贵**
  - 问题：在大规模问题（如 LLM）上，二阶优化的时间和显存开销难以承受
- **PPO（Proximal Policy Optimization，2017）**：TRPO 的工程友好版
  - 思路：不用二阶优化精确求解信任域，而是用**裁剪（Clipping）做一阶近似**
  - 牺牲：理论上不如 TRPO 严谨（裁剪是启发式的，不是严格约束）
  - 收获：计算简单、工程实现容易、调参更直觉 → 成为工业界默认选择
- **类比**：TRPO 像一个严格的法律条文（精确但复杂），PPO 像一个务实的经验法则（近似但好用）
- **你将理解**：PPO 的 clip 不是一个孤立的技巧，而是对 TRPO 信任域思想的简化。理解了 TRPO 的"为什么需要信任域"，就能理解 PPO 的"clip 是怎么近似信任域的"

## 5.5 PPO 的裁剪机制（Clipping）

- PPO 的核心目标函数：L = min( r_t × A_t, clip(r_t, 1-ε, 1+ε) × A_t )
- 裁剪的效果：
  - 当优势 A_t > 0（好动作）：r_t 被限制不超过 1+ε，防止"好的就贪心过头"
  - 当优势 A_t < 0（坏动作）：r_t 被限制不低于 1-ε，防止"坏的就矫枉过正"
- 为什么 min 操作能防止破坏性更新？——取更保守的那个估计
- ε 的典型值：0.1~0.2，过大则更新太保守，过小则保护不足
- **你将理解**：PPO 的核心思想就一句话——"更新可以，但别更新太猛"

## 5.6 GAE：广义优势估计

- 问题：如何准确估计优势函数 A(s,a)？
- 时序差分（TD）估计：偏差大、方差小
- 蒙特卡洛（MC）估计：偏差小、方差大
- GAE = 在两者之间的连续插值：Â_t = Σ (γλ)^k δ_k
- λ 的意义：λ=0 退化为 TD（高偏差），λ=1 退化为 MC（高方差），通常取 0.95
- **你将理解**：偏差-方差权衡是 RL 中无处不在的核心张力

## 5.7 回看实践：分析 LunarLander 训练日志

- 找出 clip fraction 过大的时刻 → 对应 reward 跳水
- 调整 ε 和 λ，观察稳定性变化：
  - ε 从 0.2 降到 0.1 → 更新更保守，训练更稳定但可能更慢
  - λ 从 0.95 升到 0.99 → 方差增大，曲线更震荡

## 5.8 奖励模型：RLHF 的"人工裁判"

- **为什么需要 Reward Model？**：游戏环境自动给出分数（+1/-1），但 LLM 生成的回答没有自动评分器——"这个回答好不好"需要人类来判断
- **两条截然不同的强化路径**：
  - **主观对齐（如礼貌、安全、无害）**：客观上没有绝对对错。必须通过人类偏好训练一个 Reward Model 当裁判，走标准的 RLHF 路径。
  - **客观推理（如数学、代码）**：有明确规则校验（答案正确与否、代码是否编译通过）。此时可以**完全抛弃 Reward Model，甚至抛弃 SFT 冷启动**，直接用代码规则（Rule-based Reward）对 Base 模型进行纯粹的强化学习（Pure RL，如 DeepSeek-R1-Zero）。
- **本章重点：主观对齐下的偏好收集**：给标注员看同一个问题的两个回答，让他们选"哪个更好"（比较式标注，而非打分式标注）
- **Bradley-Terry 模型**：把"成对比较"转化为"绝对分数"
  - P(y_w > y_l | x) = σ(r(x, y_w) - r(x, y_l))
  - 直觉：两个回答的分数差越大，高分者被选中的概率就越接近 1
- **训练 RM**：用一个预训练语言模型作为骨干，输出一个标量分数 r(x, y)，用偏好对做监督学习
- **RM 的三大痛点**：
  1. **标注成本极高**：成千上万的偏好对，需要大量人工标注
  2. **容易被 hack**：模型可能学会"说长话""格式好看"而非"真正有道理"
  3. **分布漂移**：RM 是在旧数据上训练的，策略更新后生成的回答分布变了，RM 的评分可能不再可靠
- **你将理解**：在主观对齐任务中，PPO 需要一个庞大且脆弱的 RM 当裁判。**这恰恰是下一章 DPO 要绕过的核心瓶颈，也是第 8 章 Pure RL 范式试图彻底抛弃的包袱。**

## 5.9 稀疏奖励与信用分配：大模型 RL 的核心难题

- **大模型生成的特殊性**：LLM 生成一个回答，可能需要输出 500 个 token——这相当于 RL 中的 500 步连续动作。但 Reward Model 只在**最后一个 token** 处给出一个评分
- **这就是最极端的稀疏奖励问题**：500 步连续决策，1 个奖励信号。"我这 500 个词里，到底是哪几个词写得好，才让我拿到了高分？"
- **信用分配（Credit Assignment）问题**：
  - 奖励是给整个序列的，但梯度更新需要分解到每个 token
  - 我们需要知道每个 token 对最终得分的**贡献度**
- **PPO 的信用分配方案**：
  - 计算每个 token 的 log π(a_t|s_t)（对数概率）
  - 用优势函数 A_t 衡量"这个 token 比平均水平好多少"
  - 更新公式：∇L ∝ A_t × ∇log π(a_t|s_t)——**只强化那些"真正有帮助"的 token**
  - KL 散度惩罚：在 token 级别约束策略不能偏离太远，防止"为了追高分而胡说八道"
- **为什么 GRPO 也能做信用分配？**（预览第 8 章）GRPO 用组内归一化的优势来替代 Critic 的估计，本质上也是在做 token 级别的信用分配——"在这 k 个回答中，当前位置选这个词的比选那个词的效果更好"
- **你将理解**：理解了稀疏奖励和信用分配，你就能明白为什么 PPO 需要 Token 级别的 log prob 和 KL 惩罚，以及为什么 GRPO 能在长文本推理（如 DeepSeek-R1 的思考过程）中发挥奇效

## 5.10 PPO 在 LLM 对齐中的完整图景

- Actor = 语言模型（生成回答）
- Critic = 价值模型（评估回答的好坏）
- 裁剪防止语言能力崩坏——不会因为一次 RL 更新就"忘了怎么说中文"
- KL 散度惩罚 = 额外的安全带，防止模型偏离原始分布太远
- 四个模型同时跑的壮观场面：Actor + Critic + Reference + Reward Model
- **你将理解**：游戏环境中的 PPO 和 LLM 对齐中的 PPO，核心机制完全一致——但 LLM 场景需要额外训练一个 Reward Model，这正是工程上最沉重的负担
