# 附录C：算法选型指南

> **本附录目标**：帮助你在面对实际问题时，快速选择合适的 RL 算法。不是追求"最强算法"，而是找到"最合适的算法"。
>
> **分水岭**：在第 3-8 章中，我们走的是一条 **Policy-Based（策略优先）** 的现代路线——从策略梯度到 Actor-Critic，从 PPO 到 DPO 再到 GRPO。这是当今大模型（生成式 AI）对齐的核心范式。但在游戏、自动驾驶和机器人领域，还有另一条极其经典的 **Value-Based（价值优先）** 路线——用价值函数直接指导决策，不需要显式的策略网络。接下来让我们进入第 4 章的 DQN，看看这条路线的威力。

## C.1 决策速查表

| 场景                  | 推荐算法        | 核心理由                    | 不推荐                  |
| --------------------- | --------------- | --------------------------- | ----------------------- |
| 游戏控制（离散动作）  | PPO / DQN       | 环境反馈密集，样本充足      | REINFORCE（方差太大）   |
| 游戏控制（连续动作）  | PPO / TD3 / SAC | 需要精确的连续控制          | DQN（无法处理连续空间） |
| 小模型快速对齐        | DPO             | 实现简单，显存友好          | PPO（杀鸡用牛刀）       |
| 推理增强（数学/代码） | GRPO            | 规则奖励可验证，Critic 可省 | DPO（缺乏在线探索）     |
| 复杂人类偏好对齐      | PPO             | 奖励模型精细，上限更高      | DPO（受静态数据限制）   |
| 多模态 RL             | GRPO / PPO      | 需要组内比较处理异构输入    | DPO（数据标注成本高）   |

## C.2 按维度选择的思考框架

- **动作空间**：离散 → 优先考虑 DQN/PPO；连续 → 优先考虑 PPO/SAC/TD3
- **奖励信号**：密集 → PPO/DQN 都行；稀疏 → 需要奖励塑形或好奇心驱动
- **样本效率**：on-policy（PPO）效率低但稳定；off-policy（SAC/TD3）效率高但工程复杂
- **交互限制 (Online vs Offline RL)**：
  - **允许在线试错 (Online)**：在虚拟环境或大模型推理任务中，试错成本低，优先选 PPO/GRPO。
  - **只能用历史数据 (Offline)**：在真实工业推荐系统、医疗决策、自动驾驶中，在线试错可能导致严重后果或成本极高。此时只能用历史日志数据训练，即**离线强化学习 (Offline RL)**。DPO 本质上就是一种在语言模型上极度成功的 Offline 算法。在非 LLM 领域，通常选用保守策略（如 CQL）来防止模型对没见过的状态产生盲目的高估（OOD 错误）。
- **计算资源**：显存充足 → PPO 全套；显存紧张 → GRPO 或 DPO

## C.3 常见错误

- 用 DQN 处理连续动作空间（DQN 只支持离散动作）
- 在数据量不足时用 PPO（on-policy 需要大量采样）
- 用 DPO 做需要在线探索的任务（DPO 只能用固定的偏好数据集）
- 忽视环境特性（确定性 vs 随机性、稀疏奖励 vs 密集奖励）直接选"最强算法"
- **你将获得**：避免"什么都用 PPO"或"什么都用 DPO"的选型陷阱

## C.4 现代 RL 框架的工程解构

- 你在第 2 章用 `DPOTrainer` 50 行代码跑通了 DPO。但背后的框架（HuggingFace TRL）做了什么？如果你去看 TRL、OpenRLHF、veRL 等框架的源码，会发现它们都遵循相似的三层抽象：

### 三层架构：Rollout → Scoring → Training

```
┌─────────────────────────────────────────────────────┐
│                   Training Loop                      │
│   (协调器：驱动整个训练循环)                          │
│                                                      │
│  ┌──────────┐    ┌──────────┐    ┌──────────────┐   │
│  │ Rollout  │ →  │ Scoring  │ →  │  Gradient    │   │
│  │ (生成)   │    │ (评估)   │    │  Update      │   │
│  │          │    │          │    │  (梯度更新)   │   │
│  └──────────┘    └──────────┘    └──────────────┘   │
│       ↑                                │             │
│       └────────────────────────────────┘             │
└─────────────────────────────────────────────────────┘
```

- **Rollout（生成层）**：用当前策略模型（Actor）对一批 prompt 生成回答（或用环境交互收集轨迹）
  - 对应代码：`model.generate()` 或 `env.step()`
  - 关键问题：如何高效批量生成？如何管理显存？（vLLM 加速、Ray 分布式）
- **Scoring（评估层）**：对生成的回答打分
  - PPO/GRPO：用 Reward Model 打分 + Critic 估计基线（或 GRPO 用组内均值）
  - DPO：不需要这一步（损失函数直接用偏好对计算）
  - 关键问题：如何防止 reward hacking？如何做 token-level 信用分配？
- **Gradient Update（梯度更新层）**：根据评分计算 loss 并反向传播
  - PPO：clip loss + value loss + entropy bonus
  - DPO：偏好分类 loss
  - GRPO：clip loss + 组内归一化优势
  - 关键问题：如何管理多个模型（Actor、Critic、Reference、RM）的显存？

## C.5 算法谱系拓展：基于模型的强化学习 (Model-Based RL)

- **当前路线回顾 (Model-Free RL)**：从 DQN 到 PPO，从 DPO 到 GRPO，所有算法都是“无模型”的——智能体不知道环境内部是如何运作的，只能通过不断试错（采样）来积累经验。
- **什么是 Model-Based RL (MBRL)？**
  - 先学习一个“世界模型”（World Model），预测给定当前状态和动作，环境的下一个状态和奖励是什么。
  - 然后，智能体在这个**自己脑海里构建的世界模型中进行推演和规划（Planning）**，而不需要在真实环境中真的去走一步。
- **代表性工作：AlphaZero / MuZero / Dreamer**
  - MuZero：学会在不知道规则的情况下下围棋、玩 Atari。
  - Dreamer 家族：在潜空间（Latent Space）中构建世界模型，在脑内进行想象和训练，极大地提高了样本效率。
- **为什么大模型领域较少提 MBRL？**：因为大语言模型本质上**本身就是一个关于语言的世界模型**。当你要求它用思维链（CoT）进行多步推理时，它其实就是在做某种形式的“内部规划”（Internal Planning）。但在物理世界（机器人控制）中，显式的世界模型（如视频生成模型 Sora）与 RL 的结合，正成为下一代具身智能的核心方向。
- **你将思考**：未来，是否可以用 Sora 这样的世界模型，为大模型的强化学习提供无穷无尽的高质量虚拟数据？
